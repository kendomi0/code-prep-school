Time Complexity Basics

Time complexity: How many operations a piece of code performs.

Let’s take a look at time complexity. Understanding time complexity is important because writing your code to run as quickly as possible (this is called optimization) is essential. Code that takes longer to run is prone to loading issues, especially if multiple users are running that code simultaneously.

There are three different measures of time complexity: Worst-case, average-case, and best-case.

Worst-case time complexity represents the maximum number of operations a code could perform. In some cases, the worst-case time complexity is the same as the average-case and/or best-case time complexity. In some other cases, the worst-case time complexity differs from those cases.

When you write code, you want to focus on the worst-case time complexity. However, it's helpful to be aware of what average-case and best-case means.

Average-case time complexity represents the average number of operations a code could perform.

Best-case time complexity represents the minimum number of operations a code could perform.

Let's look at a simple algorithm to understand the concepts of worst-case, average-case, best-case, and notations in practice.

            
    registered_users = ["Anna Connell", "Jenny Maxwell", "Bradley Newsom"]
    inputted_user = input("Enter your full name: ") # Prompt to enter user's name
    found = False # Initialize a variable to store whether the user has been found
    for user in registered_users: # Loop through list of users
        if user == inputted_user: # Check to see if the current user matches the inputted user
            found = True # Change found flag to true if they match
            print("User is already registered")
            break # Terminate loop if found
    if not found: # If the user was not found (found is false), print this message
        print("User has not yet registered")

This code iterates through a list to find out if a user has already been registered with the system. This form of searching is called linear search. To read more about this specific type of algorithm, check out the Intro to Algorithms page.

In the worst-case scenario, the user has either not been registered, or their name is last in the list of registered users. This would require the code to iterate through the entire list. For this code, that would mean inputted_user is either not in the list, or equals "Bradley Newsom".

In the best-case scenario, the user is both registered and is first in the list, so the for loop only needs to run once and then it terminates. For this code, that would mean inputted_user equals "Anna Connell."

The average-case scenario is a bit broader, since it's not necessarily a specific scenario. It refers to when the inputted user would have an equal chance of being anywhere in the list. This is the most realistic scenario, because in practice, you wouldn't know if the inputted user is or is not registered, or where their name is in the list. On average, half of the users in the list would be checked before finding the inputted user.

Time complexity is about representing the algorithm's growth rate as the input size increases, so let's discuss the various types of growth.

Constant time complexity means there is no growth at all, regardless of the change in input size.

Linear time complexity means the growth rate increases linearly in relation to the input size.

Other types of time complexity function similarly: for instance, logarithmic time complexity means the growth rates increases logarithmically in relation to the input size.

We use certain notations to express a code's time complexity. For worst-case, we use Big O notation: O(value).

For example, if the time complexity is constant, it's O(1). If the time complexity is linear, it's O(n). N refers to the input size. If we have a list, and that list has n items, that's what n refers to.

For best-case, we use Omega(value), or Ω(value).

For average-case, we use Theta(value), or Θ(value).

Some resources use big O notation for all types of cases, but we'll use the precise notations here. But when referring to an algorithm's general time complexity, you can use Big O notation. When used in general, big O notation doesn't necessarily mean the algorithm's worst-case complexity; it's just the notation being used to describe the algorithm's "typical" time complexity in practice.

Imagine doing the same thing once, versus 5 times, versus 25 (52) times. That's what O(1) versus O(n) versus O(n2) means. So a code with a time complexity of O(n2) takes longer to run than a code with a time complexity of O(n), which takes longer than a code with a time complexity of O(1).

But of course, there are lots of different time complexities. There's logarithmic time complexity, which is O(logn). (When we use log in this context, it is with base 2.) That's a smaller time complexity than O(n). You can read more in depth about specific time complexities on their respective pages.

Let's return to the registered users example.

As said before, in the worst-case scenario, the user has either not been registered, or their name is at the end of the list of registered users. This would require the code to iterate through the entire list. This would result in a worst-case time complexity of O(n), for n (number of items in list) operations.

In the best-case scenario that the user is present in the list at the first position, the loop terminates after running once. This would result in a best-case time complexity of Ω(1).

And in the average-case scenario that half of all users will be checked before locating the user, this would result in a time complexity of Θ( (n+1) 2 ). However, since we drop constants when using Big O notation, we would simplify this to Θ(n), as the time complexity would still be linear. Constants are unnecessary in Big O notation because it's more about the type of growth: constant, logarithmic, linear, etcetera. The exponent matters most, not the coefficient.

In addition to time complexity, there's also space complexity, which is discussed less often.

Space complexity refers to the amount of memory space an algorithm occupies. Space complexity is a similar concept to time complexity, but time complexity is more important in writing efficient code.
